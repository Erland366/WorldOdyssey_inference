# Troubleshooting

## FastVideo VSA Reports Missing `to_gate_compress`

Cause: FastVideo's VSA runtime block includes `to_gate_compress`, even though the public Diffusers model metadata can still look like `WanPipeline` with `WanTransformer3DModel`.

Fix: Add zero-initialized `blocks.*.to_gate_compress.weight` and `blocks.*.to_gate_compress.bias` tensors to the tiny transformer safetensors.

## FastVideo VSA Rejects Tiny Attention Dimensions

Cause: `VIDEO_SPARSE_ATTN` supports specific head dimensions, not arbitrary small debug dimensions.

Fix: Use a supported head dimension such as `64` or `128`.

## Tiny VSA Repos Load With The Wrong FastVideo Config

Cause: FastVideo exact-matches known upstream model ids. A local or personal Hugging Face tiny repo may fall back to a generic Wan config.

Fix: Pass the required runtime config explicitly, for example `pipeline_config={"flow_shift": 5.0}`.

## FastVideo Cannot Find Safetensors After Saving

Cause: On the shared filesystem, direct file access can work before `glob()` or `iterdir()` visibility catches up. FastVideo discovers component weights by globbing.

Fix: Remove stale output directories before saving, refresh component directories after saving, and validate expected safetensors paths before running FastVideo.

## FastVideo Or Triton Picks The Wrong Compiler

Cause: Miniconda compiler wrappers can conflict with Triton or FastVideo kernel compilation.

Fix: Run FastVideo backend checks with explicit system compilers while keeping the virtualenv's Python first on `PATH`:

```bash
source .venv/bin/activate
export CC=/usr/bin/gcc
export CXX=/usr/bin/g++
export PATH="$VIRTUAL_ENV/bin:/usr/bin:$PATH"
python -m pytest tests/backends -m slow -q -s
```

For one-line validation:

```bash
source .venv/bin/activate && export CC=/usr/bin/gcc CXX=/usr/bin/g++ PATH="$VIRTUAL_ENV/bin:/usr/bin:$PATH" && python -m pytest tests/backends -m slow -q -s
```

## Diffusers Video Export Falls Back To OpenCV And Fails On `libGL.so.1`

Cause: Diffusers falls back to the legacy OpenCV video backend when `imageio` or `imageio-ffmpeg` is unavailable in the active virtual environment. OpenCV then imports system `libGL.so.1`, which may be missing on headless machines.

Fix: Install and verify the recommended backend inside the activated `.venv`:

```bash
source .venv/bin/activate
python -c "import imageio, imageio_ffmpeg; print(imageio.__version__, imageio_ffmpeg.get_ffmpeg_exe())"
```

If either import fails, install through uv's pip interface:

```bash
source .venv/bin/activate
uv pip install imageio imageio-ffmpeg
```

## Diffusers Wan Test Fails Loading `spiece.model`

Cause: Older tiny Wan artifacts can contain tokenizer metadata that asks Transformers to instantiate the slow
`T5Tokenizer` path without a saved `spiece.model`. The failure appears before model execution:

```text
TypeError: not a string
```

Fix: Use the local artifact generated by `scripts/create_tiny_wan_debug_pipeline.py`, or rebuild the existing local
artifact so `tokenizer/spiece.model` is present:

```bash
source .venv/bin/activate
python scripts/create_tiny_wan_debug_pipeline.py --recipe wan2.1-t2v-1.3b --overwrite
```

The Diffusers backend test now performs this rebuild automatically when `artifacts/tiny-wan2.1-t2v-debug` is missing
`tokenizer/spiece.model`.

## CUDA Runtime Is Newer Than The Host Driver Supports

Cause: `torch.cuda.is_available()` only proves that CUDA can be discovered. Distributed or generation workloads can
still fail later if the installed CUDA runtime is newer than the NVIDIA driver supports. On this host,
`nvidia-smi` reports driver `550.127.08` with CUDA `12.4`; the failing stack used `torch==2.11.0+cu126`, which asks
for a newer CUDA runtime than this driver can execute.

Observed failure from FastVideo NCCL initialization:

```text
Cuda failure 'CUDA driver version is insufficient for CUDA runtime version'
```

The benchmark harness reproduced the same blocker for a one-step FastVideo 5B fit check on one RTX 4090:

```bash
source .venv/bin/activate
python scripts/benchmark_wan_backends.py --stage fit --case fastvideo-5b --gpus 1 --timeout-seconds 300 --poll-interval-seconds 5 --no-save-video
```

That run peaked at roughly `9550 MiB` on GPU 0 before the FastVideo worker failed while loading the transformer through
distributed/FSDP code. The benchmark classified this as `cuda_runtime_mismatch`, not OOM. The same run's preflight
also failed the tiny CUDA/cuDNN probe with `CUDNN_STATUS_NOT_INITIALIZED`.

Observed failure from Diffusers Wan generation on the same stack:

```text
RuntimeError: cuDNN error: CUDNN_STATUS_NOT_INITIALIZED
```

Fix: Align the driver and wheel runtime before treating CUDA backend failures as model or framework bugs. Use a
runtime compatible with the host driver, or upgrade the NVIDIA driver for the installed CUDA runtime.

### FastVideo On The CUDA-12.4 Stack

The local workaround is to keep FastVideo's current API but run it on the driver-compatible CUDA 12.4 Torch stack:

```text
torch==2.6.0+cu124
torchvision==0.21.0+cu124
torchaudio==2.6.0+cu124
fastvideo==0.1.7
```

Install FastVideo with `--no-deps` after pinning Torch, otherwise its dependency metadata can upgrade Torch back to a
CUDA runtime the host driver cannot execute. The repo-local `sitecustomize.py` applies
`worldodyssey_inference.fastvideo_compat.configure_fastvideo_torch_compat()` so FastVideo worker subprocesses also get
the missing `torch._dynamo.config.recompile_limit` key before importing FastVideo.

The benchmark harness also sets `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` for child workers unless the caller
already provides a value. This is required for the default-shape FastVideo 5B run on two RTX 4090s in this environment
and for the one-GPU Diffusers 1.3B fit check. Without it, affected runs can complete denoising but OOM during VAE
decode.

## SGLang Diffusion CLI Fails On `sgl_kernel` Or CUDA Runtime

Cause: The current `sglang[diffusion]` wheels can install a CUDA 13 `sglang-kernel` stack. On this machine,
`nvidia-smi` reports driver `550.127.08` with CUDA `12.4`, while `sglang-kernel==0.4.2` loads extensions that need
CUDA 13 libraries such as `libnvrtc.so.13` and `libcudart.so.13`.

Observed failure without CUDA 13 libraries on `LD_LIBRARY_PATH`:

```text
[sgl_kernel] CRITICAL: Could not load any common_ops library
ImportError: libnvrtc.so.13: cannot open shared object file
```

Observed failure when CUDA 13 libraries are added to `LD_LIBRARY_PATH` on this driver:

```text
Cuda failure 'CUDA driver version is insufficient for CUDA runtime version'
```

Fix: Do not install current unpinned `sglang[diffusion]` into the same `.venv` used for FastVideo unless you are
deliberately switching that environment to an SGLang-compatible stack. The working local path is a pinned isolated env
created by `scripts/install_sglang_diffusion.sh`:

```text
sglang==0.5.5
torch==2.8.0+cu128
sgl-kernel==0.3.16.post5
vsa==0.0.4
cuda-python==12.9.6
pytest==9.0.3
```

This stack avoids `sglang-kernel` and `nvidia-*-cu13` packages. It needs `CUDA_HOME` pointed at the venv's NVIDIA
package and a sanitized `PATH` so Triton uses `/usr/bin/ld` instead of the Miniconda linker. See
`references/sglang-diffusion.md` for the full guide.

The current SGLang backend test is intentionally marked slow and optional:

```bash
source .venv/bin/activate
python -m pytest tests/backends/test_sglang_tiny_wan.py -m slow -s
```

If `sglang` is absent, the test skips. If `sglang` is installed but the runtime stack is incompatible, the test fails
with the real `sglang generate` output.

### Failed `torch==2.6.0+cu124` Attempt

An isolated uv environment was created at `.venv_sglangcuda12` to test whether older CUDA-12 SGLang wheels can avoid a
driver upgrade. The base CUDA probe passes with:

```text
torch 2.6.0+cu124
torch cuda 12.4
conv ok (1, 32, 5, 4, 4)
```

However, SGLang Diffusion starts at the package line that pins `torch==2.8.0`. There is no
`torch==2.8.0+cu124` wheel on the PyTorch cu124 index. Forcing `sglang==0.5.5` onto `torch==2.6.0+cu124` and the
available cu124 `sgl-kernel` wheels fails before `sglang generate --help` can start:

```text
ImportError: .../sgl_kernel/.../common_ops.abi3.so: undefined symbol: _ZN3c108ListType3getERK...
```

The available cu124 `sgl-kernel` wheels under `https://docs.sglang.io/whl/cu124/sgl-kernel/` were swept from
`0.3.9+cu124` through `0.3.14.post1+cu124`; each failed against `torch==2.6.0+cu124` with the same torch C++ ABI
symbol mismatch. This means the CUDA-12 route needs either a matching `torch==2.8.0+cu124` build, a matching
`sgl-kernel` build for torch 2.6, or a container/wheel set built consistently for this driver.

### Local `sgl-kernel` Source Build Against `torch==2.6.0+cu124`

Attempt: build SGLang `v0.5.5` `sgl-kernel` with:

```bash
source .venv_sglangcuda12/bin/activate
uv pip install --no-build-isolation /tmp/sglang-v0.5.5/sgl-kernel
```

Observed blockers:

- Conda compiler/linker flags caused CMake compiler sanity checks to fail.
- CMake 4 required `-DCMAKE_POLICY_VERSION_MINIMUM=3.5` for vendored `dlpack`.
- PyTorch CMake required explicit CUDA and NVTX paths.
- The compile is long and brittle enough that it should be treated as a production build attempt, not a quick foreground debug command.

Recommendation: prefer an official SGLang container or matching prebuilt wheel set before attempting local kernel builds.
